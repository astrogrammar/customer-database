# 受講者・講座管理アプリ 要件整理（ドラフト）

## 1. アプリの目的

- 個人運営の講座における
  **受講者・講座・支払い履歴を一元管理**する
- Excel管理から脱却し、
  **検索・集計・履歴確認を容易にする**
- 利用者は**自分一人のみ**（外部公開なし）

------

## 2. 想定ユーザー

- 管理者：本人のみ
- 同時利用者：1人
- 権限管理：不要（単一ユーザー前提）

------

## 3. 管理したいデータ（基本項目）

### 3.1 受講者（Person）

- 氏名
- メールアドレス
  - 将来的な変更を考慮（履歴 or 最新のみかは検討）
- 備考（自由記述）

### 3.2 講座（Course）

- 講座名
- 開催日（または開始日）
- 金額
- 備考

### 3.3 受講履歴（Enrollment / History）

- 受講者ID
- 講座ID
- 受講日
- 支払金額
- 備考

※
「誰が・いつ・何を・いくらで受けたか」を
**履歴として保持する設計**

------

## 4. 必要な機能（最小構成）

### 4.1 データ登録

- 受講者登録（手入力）
- 講座登録（手入力）
- 受講履歴登録

### 4.2 閲覧・検索

- 受講者ごとの受講履歴一覧
- 講座ごとの受講者一覧
- 条件検索
  - 受講者名
  - 講座名
  - 年度／期間
  - 備考キーワード

### 4.3 集計（最低限）

- 受講者ごとの受講回数
- 講座ごとの開催回数／受講者数

------

## 5. 既存Excelデータのインポート（重要）

### 5.1 前提条件

- 現在、Excelで7年分のデータが存在
- 実運用上必要なのは：
  - **直近3年分：必須**
  - **最大5年分まで：あれば尚良し**
  - 5年以上前のデータは不要

### 5.2 基本方針（暫定）

- Excel → **CSV形式で書き出し**
- アプリ側で **CSV一括インポート機能**を用意
- 一件ずつの手入力は行わない

### 5.3 検討ポイント

- CSVフォーマットを事前に固定する
- インポート時の処理方針
  - 同一人物の判定基準
    （氏名＋メール？ メールのみ？）
  - 重複データの扱い
  - エラー行の扱い（スキップ or 停止）

※
**インポート設計が破綻すると全体が崩れるため、最優先検討事項**

------

## 6. 技術方針（暫定）

### 6.1 シンプル重視

- 小規模
- 長期メンテナンス不要
- 個人運用前提

### 6.2 候補

- ローカルアプリ
  - Python + SQLite
- Webアプリ（VPS上）
  - Python + SQLite
  - 軽量Webフレームワーク（例：Flask等）

※
UI・運用負荷を見て後で確定

------

## 7. セキュリティ・運用

- 外部公開なし
- 最低限の認証（自分用）
- バックアップは手動 or 定期コピーで対応

------
